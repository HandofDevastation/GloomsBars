## ¬ß2 Mask facts (VERIFIED 2026-07-18, probe series ‚Äî see HANDOFF gate 3/3b history)
- **MaskTexture renders in Midnight.** `CircleMaskScalable` atlas exists; `SetAtlas` on a MaskTexture works.
- **`CircleMaskScalable` is NOT a usable circle at button size** ‚Äî VERIFIED twice: stretched over an icon it shows rounded corners with **flattened top/bottom/left/right edges** (‚öÝ likely 9-slice scaling, hence "Scalable"; the flattening persisted even after zoom-cropping the icon, ruling out baked-border pixels as the cause). Production + probes use **bundled mask PNGs** (`Media/masks/`, 256px, alpha channel = shape; generated by script ‚Äî see git history for the generator).
- **Edge-clamp filtering bleed ‚Äî VERIFIED 2026-07-18 (cause AND cure):** a shape that
  touches its texture canvas edge renders with flattened, blurred tangents (full-bleed
  circle PNG showed it; padded art fixed it ‚Äî clean circle confirmed in QA). Cure:
  transparent padding around the shape (current art: radius 240/256) + oversize the mask
  region by 256/240 to compensate. **Rule: ALL mask/glow art must carry transparent edge
  padding.**
- **Measured on `ActionButton1`** (VERIFIED 2026-07-18, Jason's UI scale): `.icon` 45.0√ó45.0, `.IconMask` **64.0√ó64.0** (larger than the icon ‚Äî so it does NOT clip the icon's interior; mask-intersection theory dead), `.SlotBackground` 45.0√ó45.0.
- **Hard cap: 3 masks per texture.** `AddMaskTexture` throws `"Texture already has the maximum number of mask textures (3)"`. Blizzard uses 1 (`IconMask`) ‚Üí we have budget for 2, production uses 1. Probes must be idempotent toggles.
- **Editing an already-rendered mask does NOT re-render** ‚Äî `SetAtlas` on Blizzard's live `IconMask` changed `GetAtlas()` but never changed pixels, even after `RemoveMaskTexture`+`AddMaskTexture`. A **freshly created** `CreateMaskTexture` + `AddMaskTexture` renders immediately. Production: always create our own masks; never mutate Blizzard's.
- **Masks persist** through Blizzard's update cycle (mouseover etc.).
- **`SetVertexColor` on `.icon` is stomped** on mouseover (Blizzard's range/usability tint path) ‚Äî color/texcoord styling needs `hooksecurefunc` re-assertion.
- **Pressing a button re-shows `NormalTexture`** (VERIFIED 2026-07-18: hidden border art came back on press ‚Äî Blizzard's button-state machinery). Suppressing border/slot art needs re-assert hooks, same as colors.
- Icons have baked-in square border art ‚Üí **zoom-crop (`SetTexCoord ~0.08‚Äì0.92`) before masking** so a shaped mask's edge lands on artwork, not border pixels.
- Cooldown swipe + Blizzard's countdown number render normally on a styled (masked/zoomed) icon; swipe *shape* on round icons not yet assessed (too much border clutter in the test ‚Äî revisit after the skin suppresses art cleanly).

## ¬ß3 Hook points ‚Äî SOURCE-VERIFIED (wow-ui-source `live` branch @ **12.0.7 build 68453**, cloned 2026-07-18; in-game verification pending where noted)

From `Interface/AddOns/Blizzard_ActionBar/` (`Shared/ActionButton.lua` unless noted):
- **`UpdateButtonArt`** (Mainline `ActionButtonOverrides.lua` overrides the empty Shared base) ‚Äî the ONLY Lua path that re-shows `SlotArt`/`SlotBackground` and re-sets Normal/Pushed atlases. Branches on `self.bar.hideBarArt` (Edit Mode per-bar "hide bar art"): art shown ‚Üí `SlotArt` + `UI-HUD-ActionBar-IconFrame` 46√ó45; art hidden ‚Üí `SlotBackground` + `UI-HUD-ActionBar-IconFrame-AddRow` 51√ó51. Called from `BaseActionButtonMixin_OnLoad` (:1505). `SmallActionButtonMixin:UpdateButtonArt` variant sets 35√ó35. **Skin re-assert = per-button `hooksecurefunc(btn, "UpdateButtonArt", ...)`.**
- **Press border re-show is NOT Lua** ‚Äî no Lua re-shows `NormalTexture` on press; it's the C-side Button state machine (toggles Show/Hide, never alpha). **Suppress border art with `SetAlpha(0)`, never `Hide()`.** (In-game verified: alpha-0 pending; Hide() failing = verified.)
- **No `SetTexCoord` on `.icon` anywhere** in Blizzard_ActionBar ‚Üí zoom crops persist. (Consistent with in-game observation.)
- **`UpdateUsable`** (:679) owns icon vertex color: `1,1,1` usable / `0.5,0.5,1` out-of-mana / `0.4,0.4,0.4` unusable; `:Update` also `SetDesaturated`. Pure skin leaves this alone ‚Äî it's Blizzard's usability feedback.
- **HotKey**: font face/size never re-set by Blizzard (template FontObject only) ‚Üí `SetFont` sticks. Color IS stomped by the range path (:1278‚Äì89, `RED_FONT_COLOR` ‚Üî `ACTIONBAR_HOTKEY_FONT_COLOR`) ‚Äî custom hotkey *colors* need a hook; custom *fonts* don't.
- **Proc glows route through a central manager**: `ActionBarActionButtonMixin:UpdateSpellAlert` (:877) ‚Üí `ActionButtonSpellAlertManager:ShowAlert(self)` / `:HideAlert(self)`, driven by `C_SpellActivationOverlay.IsSpellOverlayed(id)`. **The glow phase hooks the manager ‚Äî one hook, not 96.**
- Mixin methods are per-frame copies ‚Üí always `hooksecurefunc` the BUTTON instances, never the mixin tables (GloomsAuras learning, holds here).
- **Cooldown path never re-sets swipe textures**: `ActionButton_UpdateCooldown` ‚Üí `ActionButton_ApplyCooldown` only calls `SetCooldown`/`Clear` (:823‚Äì868); `SetSwipeColor` appears only in cast-anim paths (:1211 hides, :1731 restores `0,0,0,1`) and per-bar OnLoads. One-time `SetSwipeTexture` on the widgets persists (‚öÝ in-game verify). Note `chargeCooldown` is edge-only by default ‚Äî leave it untouched when shaping swipes. Swipe rendering respects the swipe texture's alpha ‚Üí circular 0.8-alpha texture = round Blizzard-like sweep.
